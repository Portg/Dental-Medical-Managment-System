# SPEC-DISCOVERY.md — SPEC 对话生成流程（口腔医疗管理系统）

## 概述

通过结构化对话，让 AI 充分理解业务意图后再生成 spec，
而不是拿到需求直接写代码。

---

## Phase 1：AI 追问（暴露业务边界）

### 目标
通过 10 轮追问，确保需求的完整性和边界清晰度。

### 追问模板

```
请回答以下问题，帮助我理解需求的完整业务场景：

1. 这个功能面向哪些角色？（医生/护士/前台/管理员/药房）
2. 正常流程是什么？（从触发到结束的完整路径）
3. 异常流程有哪些？（患者取消、系统错误、并发冲突）
4. 涉及哪些现有实体？（Patient, Appointment, Invoice, MedicalCase, LabCase 等）
5. 是否涉及状态流转？（如果是，列出所有可能的状态变化）
6. 是否涉及金额计算？（折扣、退款、会员积分）
7. 数据敏感级别？（患者 NIN=高敏，病历=中敏，预约=低敏）
8. 是否需要审批流程？（折扣审批、病历修改审批、退款审批）
9. 是否影响现有功能？（哪些 Controller/Service/Model 需要修改）
10. 国际化要求？（需要添加哪些翻译 key）
```

### Phase 1 输出
- 需求边界清单
- 涉及的实体和关系图
- 潜在风险列表

---

## Phase 2：AI 扮演破坏者（挖 Anti-Goals）

### 目标
找出系统不应该做的事情，防止 AI 自行发明规则。

### 破坏者 Prompt

```
现在请扮演一个「试图破坏系统的用户」，回答：

1. 如果我是一个恶意前台，我能用这个功能做什么坏事？
   例：给自己的朋友开大额折扣、修改已完成的病历
2. 如果两个医生同时操作同一个患者，会发生什么？
3. 如果网络中断恢复后，数据会不一致吗？
4. 如果输入极端值（金额=0、数量=999999、日期=过去），会怎样？
5. 删除/软删除后，关联数据会怎样？

---
以下是本系统已知的业务约束，请核对是否遗漏：

- BR-035: 折扣 > 500 元需要管理员审批
- 退款 > 100 元需要审批
- 病历锁定后修改需要 Amendment 审批
- 预约同一时段同一椅位不能重复预约
- 患者合并后原记录标记为 merged
- 会员余额不能为负
```

### Phase 2 输出
- Anti-Goals 清单（每条对应一个「不允许」规则）
- 并发场景分析
- 边界值测试用例

---

## Phase 3：AI 扮演合规审计员（审查数据安全）

### 目标
确保医疗数据处理符合安全规范。

### 审计 Prompt

```
现在请扮演「数据合规审计员」，审查：

1. 哪些字段属于患者隐私数据？
   - NIN（身份证号）→ 加密存储（EncryptsNin trait）
   - 病历内容 → 仅授权医生可查看
   - 联系方式 → 候诊大屏脱敏显示（masked_patient_name）

2. 日志中是否会泄露敏感信息？
   - 不得记录完整 NIN
   - 不得记录患者病历详情
   - 不得记录会员密码

3. 数据修改是否可追溯？
   - 病历实现 AuditableContract（完整变更追踪）
   - 关键操作记录 `_who_added` 字段

4. 权限控制是否充分？
   - 角色：Super Administrator, Administrator, Doctor, Nurse, Receptionist
   - 每个功能点是否有 `->middleware('can:permission-slug')`？

5. API 安全
   - Sanctum token + SPA cookie 双模式认证
   - 是否所有 API 端点都在 auth:sanctum 中间件内？
```

### Phase 3 输出
- 数据分级清单（高敏/中敏/低敏）
- 权限矩阵（角色 × 功能）
- 安全风险评估

---

## Phase 4：生成 intent.md

### 综合 Phase 1~3 的输出，生成 intent.md

```markdown
# intent.md — [功能名称]

## 业务意图
[一段话描述这个功能要解决什么问题]

## 目标用户
[角色列表及其使用场景]

## 正常流程
[步骤列表]

## Anti-Goals（不允许）
- AG-001: [描述]
- AG-002: [描述]
...

## 数据敏感级别
| 字段 | 级别 | 处理方式 |
|------|------|---------|

## 依赖的现有实体
[列出涉及的 Model 及需要修改/新增的字段]

## 涉及的状态机
[引用 state-machines.yaml 中的状态图]
```

---

## 使用注意事项

1. Phase 1~3 可以在同一次对话中完成
2. intent.md 由 AI 起草、人工审定
3. 审定后的 intent.md 是后续所有 Agent 的最高优先级输入
4. 如果开发中发现 intent.md 遗漏，更新 Scratchpad 并提出变更建议
