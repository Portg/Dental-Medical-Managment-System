## Usage

### How to Invoke This Skill

```
/spec-discovery <需求描述>
```

### What This Skill Does

通过 3 个阶段的结构化对话，确保 AI 充分理解业务意图后再生成规格文件：
1. **Phase 1: 追问** — 10 轮追问暴露业务边界
2. **Phase 2: 破坏者** — AI 扮演恶意用户挖掘 Anti-Goals
3. **Phase 3: 审计员** — 审查数据安全与权限控制

最终输出 `intent.md` 更新建议。

### Common Use Cases

| 场景 | 用法 |
|------|------|
| 新功能需求 | `/spec-discovery 添加患者预约短信提醒功能` |
| 复杂流程 | `/spec-discovery 实现技工单质量管理和返工流程` |
| 规则变更 | `/spec-discovery 修改折扣审批规则为分级审批` |
| 模块重构 | `/spec-discovery 重构收费模块支持分期付款` |

## Phase 1: AI 追问（暴露业务边界）

### 向用户提出以下问题

1. **角色** — 这个功能面向哪些角色？（医生/护士/前台/管理员/药房）
2. **正常流程** — 从触发到结束的完整路径是什么？
3. **异常流程** — 患者取消、系统错误、并发冲突时怎么处理？
4. **涉及实体** — 涉及哪些现有实体？（Patient, Appointment, Invoice, MedicalCase, LabCase 等）
5. **状态流转** — 是否涉及状态变化？列出所有可能的状态
6. **金额计算** — 是否涉及折扣、退款、会员积分等财务操作？
7. **敏感数据** — 涉及哪些敏感字段？（NIN=高敏，病历=中敏，预约=低敏）
8. **审批流程** — 是否需要审批？（折扣审批、病历修改审批、退款审批）
9. **影响范围** — 是否影响现有功能？哪些 Controller/Service/Model 需要修改？
10. **国际化** — 需要添加哪些翻译 key？

### Phase 1 输出
- 需求边界清单
- 涉及的实体和关系
- 潜在风险列表

## Phase 2: AI 扮演破坏者（挖 Anti-Goals）

### 自动执行以下分析（不需要用户输入）

基于 Phase 1 收集的信息，AI 扮演「试图破坏系统的用户」思考：

1. **权限滥用** — 如果某个角色恶意操作，能做什么坏事？
   例：前台给朋友开大额折扣、医生修改他人病历
2. **并发冲突** — 两个用户同时操作同一资源会怎样？
3. **网络异常** — 中断恢复后数据是否一致？
4. **极端输入** — 金额=0、数量=999999、日期=过去
5. **删除级联** — 软删除后关联数据是否正确处理？

### 对照现有 Anti-Goals 检查遗漏

读取 `ai-dev-template/ai-spec/domain/intent.md` 中已有的 AG-001~AG-020，检查：
- 新需求是否会与现有 Anti-Goals 冲突？
- 是否需要新增 Anti-Goals？

### Phase 2 输出
- 新增的 Anti-Goals 建议（编号接续现有的）
- 并发场景分析
- 边界值测试用例建议

---

## Phase 3: 数据安全审查

### 自动执行以下检查

1. **数据分级** — 新增字段的敏感级别是什么？
2. **日志安全** — 是否有敏感字段可能出现在日志中？
3. **变更追溯** — 关键操作是否需要 Auditable？
4. **权限控制** — 每个操作是否需要 `->middleware('can:...')`？
5. **API 安全** — API 端点是否在 `auth:sanctum` 内？

### Phase 3 输出
- 数据分级建议
- 权限矩阵（角色 × 操作）
- 安全风险提醒

---

## Phase 4: 生成 intent.md 更新建议

综合 Phase 1~3，输出 intent.md 更新内容：

```markdown
## intent.md 更新建议

### 新增/修改的业务意图
[描述]

### 目标用户
[角色 × 场景]

### 正常流程
1. [步骤]
2. [步骤]
...

### 新增 Anti-Goals
- AG-0XX: [描述]
- AG-0XX: [描述]

### 数据敏感级别（新增字段）
| 字段 | 级别 | 处理方式 |
|------|------|---------|

### 涉及的实体变更
| 实体 | 变更类型 | 说明 |
|------|---------|------|

### 涉及的状态机变更
[引用/新增 state-machines.yaml 中的定义]

### Rule Key 变更
[引用/新增 rule-engine.md 中的规则]
```

**注意**：intent.md 由 AI 起草、用户审定。审定后即为后续所有 Agent 的最高优先级输入。

## 核心原则

- **追问优先** — 不确定的事情先问，不要替用户做假设
- **Anti-Goals 驱动** — 系统「不应该做什么」比「应该做什么」更重要
- **渐进式** — 每个 Phase 结束时暂停，等待用户确认后再继续
- **可追溯** — 所有发现记入 SCRATCHPAD.md，便于后续 Agent 查阅
